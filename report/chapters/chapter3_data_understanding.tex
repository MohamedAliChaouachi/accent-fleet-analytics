\chapter{Data Understanding through PostgreSQL Profiling}
\label{chap:data}

\section{Introduction}
The previous chapter has defined the architectural target and the technology stack. The present chapter executes the second phase of the CRISP-DM methodology: the \emph{data understanding}. The chapter describes the operational PostgreSQL database that supplies the project, formalizes the extraction protocol, presents the schema of the staging layer through a data dictionary, audits the quality of the data with explicit SQL queries, profiles the missing values, and concludes with the exploratory data analysis that informs the feature engineering of Chapter~\ref{chap:prep}. All scripts referenced here are listed in Appendix~\ref{app:sql} and are available in the repository under \texttt{scripts/sql/}.

\section{Source database presentation}

\subsection{Operational PostgreSQL of \hostorg}
The source database is a PostgreSQL~16 instance hosting the operational data of the telematics platform. The instance is reachable from the analytical Azure VM through a secured connection. The schema of interest comprises about a dozen tables, the most voluminous of which are the trip table (\texttt{path}) and the telemetry archive (\texttt{archive}), with respectively 7.4~million and 54.7~million rows over the available history.

\subsection{Tables of interest}
Five families of tables have been retained for the project: the trip-related family (\texttt{path}, \texttt{rep\_overspeed}, \texttt{stop}), the alert family (\texttt{notification}), the daily aggregate family (\texttt{rep\_activity\_daily}), the telemetry family (\texttt{archive}) and the master-data family (\texttt{vehicule}, \texttt{driver}, \texttt{maintenance}, \texttt{document}, fueling tables).

\section{Extraction strategy}

\subsection{Connection setup}
The connection to the source database follows the standard Python toolchain (\texttt{psycopg2} or SQLAlchemy). The connection parameters are externalized in a \texttt{.env} file, with credentials of a read-only service account dedicated to analytical extraction. Listing~\ref{lst:psycopg-connect} shows the canonical connection snippet.

\begin{lstlisting}[style=python,caption={Read-only connection from Python to PostgreSQL.},label={lst:psycopg-connect}]
import os
from sqlalchemy import create_engine

PG_DSN = (
    f"postgresql+psycopg2://{os.environ['PG_USER']}:{os.environ['PG_PWD']}"
    f"@{os.environ['PG_HOST']}:{os.environ['PG_PORT']}/{os.environ['PG_DB']}"
)
engine = create_engine(PG_DSN, pool_pre_ping=True, future=True)
\end{lstlisting}

\subsection{MCP-driven exploration}
During the data understanding phase, the Model Context Protocol (MCP) PostgreSQL server has been used as a programmable interface for ad-hoc exploration. Each exploratory question (row counts, null rates, distinct values, distributions) is expressed as an SQL query and submitted to the MCP server, which returns a structured result that is then post-processed in the notebooks. This setup substantially accelerates the iteration cycle and standardizes the queries.

\subsection{Automated extraction to CSV}
For reproducibility purposes, the principal exploratory queries are also exported as CSV through a dedicated Python script (\texttt{scripts/python/extract\_to\_csv.py}, Appendix~\ref{app:python}). The script reads a manifest of queries, executes them and writes the results to a versioned \texttt{exports/} directory.

\section{Schema analysis}

\subsection{Schema inspection queries}
The schema of the source tables is inspected through the standard catalogue queries of PostgreSQL on \texttt{information\_schema}. Listing~\ref{lst:schema-columns} shows the canonical inspection query, which retrieves, for every column of every table, its name, its data type, its nullability, its default and its ordinal position.

\begin{lstlisting}[style=sql,caption={Schema inspection query against \texttt{information\_schema}.},label={lst:schema-columns}]
SELECT
    table_schema,
    table_name,
    column_name,
    data_type,
    is_nullable,
    column_default,
    ordinal_position
FROM information_schema.columns
WHERE table_schema IN ('staging', 'public')
ORDER BY table_schema, table_name, ordinal_position;
\end{lstlisting}

\section{Data quality assessment}

\subsection{Row count profile}
A first quality indicator is the row count per table, computed by Listing~\ref{lst:row-count}. The result of the query is stored as a baseline against which subsequent runs can be compared in order to detect any abnormal contraction or expansion of the source.

\begin{lstlisting}[style=sql,caption={Row counts per table of interest.},label={lst:row-count}]
SELECT 'path' AS table_name, COUNT(*) AS rows FROM staging.path
UNION ALL
SELECT 'rep_overspeed',         COUNT(*) FROM staging.rep_overspeed
UNION ALL
SELECT 'stop',                  COUNT(*) FROM staging.stop
UNION ALL
SELECT 'notification',          COUNT(*) FROM staging.notification
UNION ALL
SELECT 'rep_activity_daily',    COUNT(*) FROM staging.rep_activity_daily
UNION ALL
SELECT 'archive',               COUNT(*) FROM staging.archive
ORDER BY rows DESC;
\end{lstlisting}

\subsection{Null-rate analysis}
For every column of every table of interest, the null rate is computed by Listing~\ref{lst:null-rate}. The null rates are then post-processed into a heatmap rendered by the Python script \texttt{fig\_missing\_heatmap.py}, reproduced in Figure~\ref{fig:missing-heatmap}.

\begin{lstlisting}[style=sql,caption={Null-rate analysis for \texttt{staging.path}.},label={lst:null-rate}]
SELECT
    'distance_driven' AS column_name,
    100.0 * SUM(CASE WHEN distance_driven IS NULL THEN 1 ELSE 0 END) / COUNT(*) AS null_pct
FROM staging.path
UNION ALL
SELECT 'max_speed',
    100.0 * SUM(CASE WHEN max_speed IS NULL THEN 1 ELSE 0 END) / COUNT(*) FROM staging.path
UNION ALL
SELECT 'path_duration',
    100.0 * SUM(CASE WHEN path_duration IS NULL THEN 1 ELSE 0 END) / COUNT(*) FROM staging.path
UNION ALL
SELECT 'fuel_used',
    100.0 * SUM(CASE WHEN fuel_used IS NULL THEN 1 ELSE 0 END) / COUNT(*) FROM staging.path;
\end{lstlisting}

\subsection{Referential integrity}
The referential integrity between the source tables and the prospective dimensions is verified by Listing~\ref{lst:integrity}. The query identifies the devices appearing in the trip table that are not linked to any vehicle through the assignment chain, and quantifies their share in the total volume.

\begin{lstlisting}[style=sql,caption={Referential integrity between \texttt{staging.path} and the device--vehicle chain.},label={lst:integrity}]
SELECT COUNT(DISTINCT p.device_id) AS orphan_devices,
       SUM(CASE WHEN v.vehicule_id IS NULL THEN 1 ELSE 0 END) AS orphan_rows
FROM staging.path p
LEFT JOIN staging.assignment a USING (device_id)
LEFT JOIN staging.vehicule  v ON v.vehicule_id = a.vehicule_id;
\end{lstlisting}

\subsection{Outlier detection}
Outliers on the physical bounds (negative distances, durations, speeds above 200~km/h) are detected by Listing~\ref{lst:outliers}. The result of this query motivates the cleaning rules \texttt{C2}, \texttt{C3} and \texttt{C5} formalized in Chapter~\ref{chap:prep}.

\begin{lstlisting}[style=sql,caption={Detection of physically implausible values in \texttt{staging.path}.},label={lst:outliers}]
SELECT
    SUM(CASE WHEN path_duration <= 0 THEN 1 ELSE 0 END) AS bad_duration,
    SUM(CASE WHEN distance_driven <= 0 THEN 1 ELSE 0 END) AS bad_distance,
    SUM(CASE WHEN max_speed > 200 THEN 1 ELSE 0 END) AS bad_speed
FROM staging.path;
\end{lstlisting}

\subsection{Bivariate analysis}
A bivariate analysis between the trip distance and the maximum speed shows a smooth growth of the maximum speed with the distance, with a saturation around 90~km/h on a clear majority of trips. A second analysis between the harsh-event ratio and the overspeed ratio reveals a separation of the operational regimes per tenant, anticipated in Chapter~\ref{chap:context}.

\subsection{Per-tenant signatures}
The aggregation of the principal indicators per tenant confirms two distinct regimes: tenant \texttt{264} exhibits an \emph{overspeed-rich} behavioural signature, with high overspeed-per-100~km values, while tenant \texttt{1787} exhibits a \emph{harsh-event-rich} signature. Tenants \texttt{235} and \texttt{238} occupy intermediate positions. This contrast directly informs the per-tenant fitting strategy retained in Chapter~\ref{chap:model}.

\subsection{Temporal coverage}
A monthly time series of trip counts and distances has been computed and plotted. The series shows a stable activity from 2025-01 onward, the period during which the archive-derived signals (harsh events, telemetry) are also available. The combination of the activity series and the archive coverage motivates the restriction of the modeling window to 2025-01--2026-04.

\textbf{ID} & \textbf{Issue} & \textbf{Source} \\

DQ-2 & Trip durations $\le 0$. & \texttt{path} \\

DQ-4 & Fuel values outside $[0,500]$~L. & \texttt{path} \\

\section{Conclusion}
This chapter has executed the data understanding phase by mobilizing the PostgreSQL profiling toolchain, by inspecting the schema, by building a data dictionary, by quantifying the data quality through dedicated SQL queries and by conducting an exploratory data analysis on the principal indicators. The eleven data quality issues identified in this chapter become, in Chapter~\ref{chap:prep}, the input specification of the cleaning rule engine that opens the data preparation phase.
