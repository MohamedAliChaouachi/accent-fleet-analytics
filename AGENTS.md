# AGENTS.md — Agent onboarding and customization

Purpose: concise instructions to help AI coding agents be productive in this repository.

Quick start
- Setup venv and deps: `python -m venv .venv` then `.
  .venv\Scripts\Activate.ps1` (Windows) or `source .venv/bin/activate` (Unix).
- Install: `pip install -r requirements.txt`.
- Common make targets: `make up` (start stack), `make down`, `make test`, `make lint`, `make train`.
- Use `cp .env.example .env` and edit PostgreSQL / MLflow credentials before running.

Project layout (top-level)
- `app/` — FastAPI service (routes, auth, middleware).
- `scripts/` — ETL & training utilities (`run_batch.py`, `train_clustering.py`).
- `sql/` — Idempotent SQL transforms, views, marts.
- `web/` — React SPA (Vite + TypeScript) — the production frontend.
- `config/` — YAML feature & pipeline configs.
- `notebooks/` — CRISP-DM notebooks (00–06).
- `tests/` — pytest suites (unit & integration).
- `mlruns/`, `models/` — MLflow and model artifacts.

What agents should know (concise)
- Build & run: prefer `make` targets listed above. Docker images are defined in `docker/` and `docker-compose.yml`.
- Tests: `make test` runs pytest; integration tests are marked `integration` and skipped unless a reachable Postgres is configured.
- Config: use `.env` (copy from `.env.example`). Avoid committing secrets.
- Conventions: pipeline uses schemas `staging`, `warehouse`, `marts`; unit-of-analysis is `device-month`.
- Gotcha: do NOT leave inline comments on numeric env values in `.env` (can break pydantic parsing).

Agent constraints & etiquette
- Preserve repository style and existing SQL/templating patterns.
- When proposing DB changes, prefer incremental, idempotent SQL and include migration/backfill notes.
- Do not modify secrets or `.env` values; suggest edits to `.env.example` instead.
- For long-running or infra changes (Docker, Azure Postgres), propose steps and tests but do not execute without human approval.
